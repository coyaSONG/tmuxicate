<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| config role metadata -> routing policy | Operator-declared role specs and priorities control which agents may receive coordinator work |
| route-task request -> owner selection | Coordinator-provided `task_class` and `domains` become deterministic owner selection inputs |
| route decision -> durable task creation | Selected owner and routing evidence are persisted into canonical child-task state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T | `internal/config/config.go` / `internal/config/loader.go` | mitigate | Validate `RoleSpec`, `route_priority`, and per-class routing policy so malformed config cannot silently widen candidate sets. |
| T-02-02 | T | `internal/session/run.go` | mitigate | Route only from the run's `allowed_owners` snapshot, require exact `TaskClass` + domain coverage, and make tie-breaking deterministic. |
| T-02-03 | R | coordinator routing outcomes | mitigate | Return structured `RouteRejection` data and persist structured `RoutingDecision` fields so routing choices are inspectable after the fact. |
| T-02-04 | I | root-message contract | mitigate | Keep the root message limited to routing instructions and durable references; do not embed hidden heuristics or secret-bearing runtime state. |
</threat_model>
