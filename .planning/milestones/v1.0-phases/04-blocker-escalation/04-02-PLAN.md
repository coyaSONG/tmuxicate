  3. defaulting `source_task_id == current_task_id` on first block/wait

Implement the exact policy table:
- `dependency_reply` -> `watch`
- `external_event` -> `watch`
- `agent_clarification` -> `clarification_request`

- `reroute_needed` with `reroute_count >= resolved ceiling` -> `escalate`
- `human_decision` -> `escalate`
- `unsupported` -> `escalate`

Automatic side effects:
- `watch`: persist/update blocker case with `status=active`, no reroute-count increment
- `clarification_request`: persist/update blocker case with `status=active`, no reroute-count increment

- `escalate`: set `status=escalated`, set `recommended_action.kind=manual_reroute` when escalating from `reroute_needed`, otherwise `recommended_action.kind=clarify`, and record `recommended_action.note`

Operator resolution in `internal/session/blocker_resolve.go`:
- require an existing `status=escalated` blocker case
- `manual_reroute`: require non-empty `reason`; reroute the original source task with `RouteChildTask`; when `--owner` is set, pass it as `OwnerOverride` with the same `reason`; record `resolution.action=manual_reroute`, `resolution.note=reason`, `resolution.created_task_id`, `resolution.created_message_id`, and `status=resolved`
- `clarify`: require `--body-file` or `--stdin`; send a `decision` message to `current_owner` with `Send(...)`, `KindDecision`, `Thread` set to the run root thread, and `ReplyTo` set to `current_message_id`; record `resolution.action=clarify`, `resolution.note=reason`, `resolution.created_message_id`, and `status=resolved`
- `dismiss`: require non-empty `reason`; record `resolution.action=dismiss`, `resolution.note=reason`, and `status=resolved` without creating new work

Do not add a human mailbox recipient model, a blocker-list command, or any run-level blocker summary section in this plan, per D-12, D-17, and D-18.
