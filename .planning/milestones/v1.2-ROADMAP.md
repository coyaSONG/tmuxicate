# Milestone v1.2: Remote Execution Foundations

**Status:** ✅ SHIPPED 2026-04-11
**Phases:** 10-12
**Total Plans:** 6

## Overview

This archive captures the third shipped coordinator milestone for `tmuxicate`: the first concrete remote execution foundation built on top of explicit target metadata, adding command-based non-local dispatch, durable target health, and explicit operator control without replacing the mailbox-backed coordination model.

## Milestone Summary

- Non-local execution targets now declare concrete dispatch and heartbeat contracts instead of existing only as placement metadata.
- Routed work can launch through a stable target command environment while preserving canonical run, task, and routing artifacts.
- Operators can inspect target readiness, heartbeat freshness, pending dispatches, and failed dispatches through both `status` and the new `target` command family.
- Routing now excludes unavailable targets with durable excluded-target evidence, and operator recovery can redispatch unread pending work explicitly.
- The milestone preserves lifecycle parity by keeping remote workers on the same mailbox/task/state event contract used by local execution.

**Accepted tech debt:**
- Remote transport is still command-based and assumes shared state access rather than authenticated transport or worker bootstrap.
- Remote lifecycle visibility still depends on workers using the canonical task/state commands instead of a dedicated remote protocol.
- Non-pane targets remain outside tmux daemon health probing; their observability comes from target artifacts and operator controls.

# Roadmap: tmuxicate

## Overview

This milestone converted explicit execution-target placement into the first usable remote execution path. `v1.1` had already introduced target metadata and non-local runtime boundaries; `v1.2` made that state actionable with concrete dispatch, durable health, and operator steering while preserving the same durable coordinator artifacts and inspection surfaces.

- [x] **Phase 10: Remote Transport Contracts** - Define the first concrete target dispatch and heartbeat contract for non-pane execution (completed 2026-04-11)
- [x] **Phase 11: Target Health & Lifecycle Parity** - Make remote targets durably observable and keep non-local lifecycle behavior compatible with existing inspection flows (completed 2026-04-11)
- [x] **Phase 12: Operator Target Control** - Add explicit operator control for target availability, recovery, and explainable reroute behavior (completed 2026-04-11)

**Success Criteria** (what must be TRUE):

  2. Timeline and summary projections remain consistent for remote and local tasks.
  3. Missing or invalid remote lifecycle data fails loudly instead of being guessed.
**Plans**: 2 plans

Plans:
