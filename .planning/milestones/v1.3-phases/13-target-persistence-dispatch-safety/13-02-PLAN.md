# Phase 13 Plan 02: Intent-First Dispatch Recovery

## Objective

Make non-pane dispatch recovery explicit and idempotent so target enable/recovery flows do not silently duplicate external work.

## Tasks

1. Persist a dispatch intent for each target/message pair before executing the configured dispatch command.
2. Add idempotency and state-transition checks so redispatch skips already dispatched work and handles pending or failed records predictably.
3. Cover `dispatchNonLocalTask` and `dispatchPendingForTarget` behavior with session tests for success, failure, existing dispatched records, and recovery paths.

## Verification

- `go test ./internal/session -count=1 -race` passes.
- `EnableTarget` redispatches only unread pending work that is safe to dispatch.
- A successful dispatch followed by a recovery path does not run the configured command a second time unless future explicit force behavior is added.
- Dispatch failure remains non-fatal to canonical run/task/message artifacts and leaves operator-visible target state.

## Guardrails

- Do not change routing semantics or task creation order beyond the minimum needed for dispatch intent safety.
- Do not introduce timeouts, authenticated transport, worktrees, or cloud integrations in this plan.
- Keep dispatch side effects inspectable through durable target records and existing status/target views.
