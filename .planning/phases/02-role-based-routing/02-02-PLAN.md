<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| route request -> duplicate key | Coordinator-provided task class/domains become a durable duplicate identity |
| duplicate scan -> task persistence | Existing task artifacts and receipt state govern whether new work may be written |
| routed task artifacts -> operator inspection | Persisted routing evidence is rendered back to humans through `run show` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | T | `internal/session/run.go` | mitigate | Normalize domains deterministically and enforce duplicate policy from `(run_id, task_class, normalized_domains)` rather than owner or message kind. |
| T-02-05 | T | duplicate scan + persistence | mitigate | Use a run-scoped lock and re-check duplicates inside `AddChildTask` so racing route requests cannot both persist the same exclusive work. |
| T-02-06 | R | `internal/protocol/coordinator.go` / `internal/session/run_rebuild.go` | mitigate | Persist routing decision evidence on canonical child-task records and render it in `run show` so route acceptance or rejection is auditable from disk. |
| T-02-07 | E | owner override path | mitigate | Require explicit override reasons, keep override targets inside teammate boundaries, and never let overrides bypass duplicate blocking. |
</threat_model>
