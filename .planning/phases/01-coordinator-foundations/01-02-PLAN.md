<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI `run` input -> canonical run artifacts | Human-provided goal and coordinator selection become durable records and mailbox messages |
| CLI `run add-task` input -> canonical child-task artifacts | Coordinator-provided task fields become persisted graph edges and recipient receipts |
| canonical coordinator artifacts -> mailbox messages | Durable run/task state must stay aligned with emitted mailbox work |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | T | `internal/session/run.go` | mitigate | Validate coordinator target, owner, dependency IDs, and required text fields before any path join or mailbox write occurs. |
| T-01-06 | T | `internal/mailbox/coordinator_store.go` | mitigate | Write `run.yaml` and task YAML atomically under deterministic paths before creating any message/receipt, preventing partial artifacts and malformed reconstruction state. |
| T-01-07 | R | coordinator artifact/message linkage | mitigate | Store `run_id`, `task_id`, `parent_run_id`, `message_id`, and `thread_id` in canonical records and root/child task message metadata so disk-only reconstruction can prove lineage. |
| T-01-08 | I | durable coordinator artifacts | mitigate | Persist only goal/task metadata, owner/team snapshot, and durable references; do not serialize env vars, transcript paths, or runtime logs into `run.yaml` or task YAML files. |
</threat_model>
